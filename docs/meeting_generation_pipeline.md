# WIAMeet 회의록 생성 프로세스 소개

이 문서는 WIAMeet이 회의 녹음 파일과 회의 기본 정보를 바탕으로 회의록을 자동 생성하는 전체 과정을 소개하기 위한 자료입니다. 코드 구조나 함수 설명보다, 사용자가 경험하는 단계와 시스템이 내부에서 수행하는 핵심 처리 흐름을 중심으로 설명합니다.

## 1. 회의 정보 입력

회의록 생성은 회의의 기본 정보를 입력하는 단계에서 시작됩니다.

사용자는 회의록 작성에 필요한 맥락 정보를 함께 입력합니다.

- 회의명
- 회의 카테고리
- 회의 목적
- 회의 일자
- 시작 시간과 종료 시간
- 회의 참석 조직
- 회의 참석자 명단
- 회의 녹음 파일
- 회의 참고자료, 선택 사항

이 정보는 단순한 저장용 데이터가 아니라, 이후 STT 교정, 화자 식별, 회의록 작성 품질을 높이기 위한 기준 정보로 활용됩니다.

예를 들어 회의 목적은 대화의 주제를 파악하는 데 사용되고, 참석자 명단은 화자 매칭의 후보군으로 사용됩니다. 회의 참고자료는 회의에서 등장할 수 있는 전문 용어, 프로젝트명, 조직명, 제품명 등을 파악하는 보조 자료로 사용됩니다.

## 2. 녹음 파일과 참고자료 업로드

사용자가 `회의록 분석`을 누르면 녹음 파일과 회의 정보가 서버로 전송됩니다.

대용량 녹음 파일의 경우 업로드 시간이 길어질 수 있으므로, WIAMeet은 사용자가 버튼을 누른 즉시 처리 화면을 먼저 보여줍니다. 이를 통해 사용자는 업로드와 분석이 시작되었다는 것을 바로 확인할 수 있습니다.

회의 참고자료가 함께 첨부된 경우, 시스템은 원본 파일을 보관하면서 동시에 텍스트 정보만 추출합니다.

지원하는 참고자료 형식은 다음과 같습니다.

- PDF
- PPTX
- PPT

참고자료에서 추출된 텍스트는 회의록이나 데이터베이스에 별도 저장하지 않고, STT 교정 단계에서만 활용됩니다.

## 3. 작업 공간 생성

업로드가 완료되면 시스템은 해당 회의만을 위한 독립적인 작업 공간을 생성합니다.

작업 공간에는 다음 정보가 분리되어 보관됩니다.

- 녹음 파일
- 회의 메타 정보
- 참고자료 원본 파일
- 중간 처리 결과
- 최종 회의록 초안

이 작업 공간은 사용자가 최종적으로 회의록을 확정하기 전까지는 임시 영역으로 유지됩니다. 사용자가 회의록을 확인하고 완료했을 때만 정식 회의록 저장소로 이동합니다.

이 구조는 처리 중 실패하거나 사용자가 최종 완료하지 않은 회의가 라운지에 노출되는 것을 방지합니다.

## 4. 오디오 분석 준비

시스템은 업로드된 오디오를 분석 가능한 형태로 변환합니다.

이 단계에서는 다음 작업이 수행됩니다.

- 오디오 파일 로드
- 전체 오디오 길이 확인
- 음성 처리를 위한 단일 채널 변환
- 분석 모델에 맞는 형태로 오디오 준비

이후 화자 분리와 STT 처리가 가능한 상태가 됩니다.

## 5. 화자 분리

회의 녹음에는 여러 사람이 번갈아 말하는 경우가 많습니다. WIAMeet은 먼저 녹음 전체를 분석해 누가 언제 말했는지를 구간 단위로 분리합니다.

화자 분리 단계의 목표는 다음과 같습니다.

- 발화 구간 탐지
- 발화 구간별 화자 번호 부여
- 동일 화자의 연속 발화 구간 정리
- 너무 짧거나 의미 없는 발화 구간 제거
- 겹치는 발화 구간 보정

예를 들어 시스템은 다음과 같은 정보를 만듭니다.

```text
Speaker 0: 10.0s - 18.5s
Speaker 1: 19.2s - 24.0s
Speaker 0: 25.1s - 32.7s
```

이 단계의 결과는 아직 실제 사람 이름이 아니라 `Speaker 0`, `Speaker 1`과 같은 임시 화자 번호입니다.

## 6. 화자 구간 전처리

화자 분리 결과는 그대로 사용하기에는 너무 잘게 나뉘거나, 일부 구간이 겹치거나, 매우 짧은 발화가 포함될 수 있습니다.

WIAMeet은 STT 품질을 높이기 위해 화자 구간을 정리합니다.

주요 전처리 방향은 다음과 같습니다.

- 같은 화자가 짧은 간격으로 이어 말한 경우 하나의 구간으로 병합
- 너무 짧은 발화 구간 제거
- STT에 넣기 적절한 길이로 구간 제한
- 겹치는 시간 구간 보정

이 과정을 통해 STT 모델이 더 안정적으로 음성을 인식할 수 있는 단위가 만들어집니다.

## 7. STT 변환

정리된 화자 구간별로 음성 인식이 실행됩니다.

각 구간은 다음과 같은 문장 단위 결과로 변환됩니다.

```text
[1] Speaker 0 (10.0s - 18.5s): 오늘 회의 목적은 데이터 플랫폼 검토입니다.
[2] Speaker 1 (19.2s - 24.0s): 비용 구조를 먼저 확인하면 좋겠습니다.
```

이 단계에서 생성되는 결과는 아직 원본 STT에 가깝습니다. 따라서 일부 단어가 잘못 받아써졌거나, 띄어쓰기와 맞춤법이 부정확할 수 있습니다.

## 8. STT 결과 교정

STT 결과가 만들어지면 LLM을 활용해 문맥 기반 교정을 수행합니다.

교정의 목적은 회의 내용을 새로 쓰는 것이 아니라, 명백한 STT 오류만 바로잡는 것입니다.

교정 대상은 다음과 같습니다.

- 맞춤법 오류
- 띄어쓰기 오류
- 조사 오류
- 오타
- 잘못 받아쓴 단어
- 회의 맥락상 명백히 다른 용어로 인식된 표현

이때 다음 정보가 보조 자료로 사용됩니다.

- 회의 목적
- 회의 참석자 명단
- 회의 참고자료에서 추출한 텍스트

특히 회의 참고자료가 있는 경우, 시스템은 이를 회의 관련 단어집처럼 활용합니다. 참고자료에 등장하는 전문 용어, 프로젝트명, 조직명, 제품명, 영문 약어 등을 기준으로 STT가 비슷한 발음의 다른 단어로 잘못 받아쓴 부분을 교정합니다.

단, 참고자료는 오직 잘못 받아쓴 단어를 바로잡기 위한 보조 자료로만 사용됩니다. 원문에 없는 내용을 추가하거나, 회의 내용을 참고자료 기준으로 재작성하지 않습니다.

## 9. 화자 자동 매칭

STT 교정이 끝나면 시스템은 `Speaker 0`, `Speaker 1`과 같은 임시 화자 번호를 실제 참석자와 연결합니다.

화자 매칭에는 다음 정보가 활용됩니다.

- 회의 참석자 명단
- 각 Speaker의 발화 내용
- 발화 맥락
- 자기소개나 직책 언급
- 특정 업무 영역에 대한 발언 패턴

예시:

```text
Speaker 0 -> 류여형 책임매니저
Speaker 1 -> DataBricks 직원 1
```

매칭 결과에는 신뢰도와 매칭 근거가 함께 제공됩니다. 신뢰도가 낮거나 근거가 부족한 경우에는 미확인 화자로 남을 수 있습니다.

## 10. 사용자 화자 매핑 확인

자동 매칭 결과는 사용자가 직접 확인할 수 있습니다.

화자 매핑 확인 화면에서는 다음 작업이 가능합니다.

- Speaker별 자동 매칭 결과 확인
- 매칭 근거 확인
- 신뢰도 확인
- 발화 샘플 확인
- 오디오 구간 재생
- 화자 이름 수정
- 발화 내용 수정
- 불필요한 발화 제거

사용자가 발화 샘플을 클릭하면 해당 시간의 오디오가 재생됩니다. 이를 통해 실제 목소리와 발화 내용을 확인하면서 화자 매핑을 검수할 수 있습니다.

이 단계는 자동 처리 결과를 사용자가 검증하는 품질 관리 단계입니다.

## 11. 회의록 작성 지시사항 입력

화자 매핑을 저장하면 회의록 생성 단계로 넘어갑니다.

사용자는 선택적으로 회의록 작성 지시사항을 입력할 수 있습니다.

예시:

```text
실장님 발언을 집중해서 정리해.
질의응답이 빠지지 않도록 정리해.
의사결정 사항과 후속 과제를 명확히 분리해.
```

이 지시사항은 회의록의 작성 관점을 조정하는 데 사용됩니다.

단, 기본 회의록 형식 자체는 WIAMeet의 회의록 생성 프롬프트 기준을 따릅니다.

## 12. 회의록 자동 생성

시스템은 최종 교정된 발화 목록과 화자 매핑 결과를 바탕으로 회의록을 생성합니다.

입력으로 사용되는 정보는 다음과 같습니다.

- 교정된 회의 발화 전체
- 실제 참석자 이름으로 매핑된 화자 정보
- 각 발화의 시간 정보
- 사용자가 입력한 작성 지시사항

회의록은 Markdown 형식으로 생성됩니다.

생성되는 회의록에는 일반적으로 다음 요소가 포함됩니다.

- 회의 개요
- 주요 논의 내용
- 결정 사항
- 후속 과제
- 확인 필요 사항

단, 실제 출력 구조는 회의록 생성 프롬프트와 사용자의 작성 지시사항에 따라 달라질 수 있습니다.

## 13. 회의록 확인과 편집

회의록이 생성되면 사용자는 생성된 Markdown 회의록을 확인합니다.

이 화면에서 사용자는 회의록을 직접 편집할 수 있습니다.

편집 가능한 이유는 다음과 같습니다.

- 자동 요약이 회의 의도와 다를 수 있음
- 특정 표현을 조직 내 문서 톤에 맞춰 조정해야 할 수 있음
- 후속 과제나 결정 사항을 사람이 최종 확인해야 함
- 회의록 배포 전 민감한 표현을 정리해야 할 수 있음

사용자는 최종본을 확인한 뒤 회의록을 확정합니다.

## 14. 최종 저장

사용자가 `완료`를 누르면 회의록이 정식으로 저장됩니다.

이 시점에 시스템은 임시 작업 공간의 파일들을 정식 저장 공간으로 이동합니다.

최종 저장되는 정보는 다음과 같습니다.

- 회의 녹음 파일
- 회의 기본 정보
- 회의 참고자료 원본 파일
- 화자 매칭 결과
- 최종 발화 목록
- 최종 회의록 Markdown 파일
- 회의록 라운지 표시용 메타 정보

최종 완료 전까지는 라운지에 노출되지 않으며, 완료 이후부터 회의록 라운지에서 조회할 수 있습니다.

## 15. 회의록 라운지 조회

완료된 회의록은 회의록 라운지에서 확인할 수 있습니다.

회의록 라운지에서는 다음 기능을 제공합니다.

- 일자별 회의록 목록 조회
- 회의 카테고리 필터링
- 회의 월 필터링
- 회의록 상세 보기
- Markdown 회의록 확인
- 회의 오디오 재생
- 발화별 회의 복기
- 회의 참고자료 목록 확인
- 참고자료 ZIP 다운로드

회의 복기 영역에서는 각 발화가 카드 형태로 표시됩니다. 발화 카드를 클릭하면 해당 시간의 오디오가 재생되어 회의 내용을 다시 확인할 수 있습니다.

## 16. Meet Home 요약

생성된 회의록 데이터는 Meet Home의 요약 지표에도 반영됩니다.

Meet Home에서는 다음 정보를 확인할 수 있습니다.

- 전체 회의록 수
- 지난주 월요일부터 금요일까지의 회의 수
- 지난주 우리 팀에서 회의 참석이 가장 많았던 사람
- 이번 달 최다 회의 조직
- 월별 회의 카테고리 분포
- 최근 회의록 목록

이 화면은 단순한 시작 페이지가 아니라, 회의록 데이터가 쌓이면서 팀의 회의 흐름을 파악하는 대시보드 역할을 합니다.

## 17. 프로세스 요약

```text
1. 회의 정보 입력
2. 녹음 파일과 참고자료 업로드
3. 작업 공간 생성
4. 오디오 분석 준비
5. 화자 분리
6. 화자 구간 전처리
7. STT 변환
8. STT 결과 교정
9. 화자 자동 매칭
10. 사용자 화자 매핑 확인
11. 회의록 작성 지시사항 입력
12. 회의록 자동 생성
13. 회의록 확인과 편집
14. 최종 저장
15. 회의록 라운지 조회
16. Meet Home 요약 반영
```

WIAMeet의 회의록 생성 프로세스는 단순히 음성을 텍스트로 바꾸는 작업이 아니라, 회의 맥락, 참석자 정보, 참고자료, 화자 검수, 최종 편집 과정을 결합해 실제 업무에 사용할 수 있는 회의록을 만드는 흐름입니다.
